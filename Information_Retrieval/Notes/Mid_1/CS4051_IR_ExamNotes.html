Chapter 1

Boolean Retrieval

1.1 What is Information Retrieval?

Definition

Information Retrieval (IR) is finding material (usually documents) of an unstructured nature (usually text) that satisfies an information need from within large collections (usually stored on computers).

The simplest retrieval is linear scanning (grepping) — reading through every document sequentially. This works for small collections but becomes impractical for large ones because:

  • Modern collections contain billions to trillions of words.
  • Flexible matching (e.g., "Romans near countrymen") is impractical to grep.
  • Ranked retrieval is not supported — grep only finds matches, not the best matches.

The solution is to index documents in advance using an inverted index.

1.2 Term-Document Incidence Matrix

The most basic data structure: a binary matrix where rows = terms, columns = documents, and a cell is 1 if the term appears in the document, 0 otherwise.

Problem: This matrix is extremely sparse (mostly 0s) and impractical to store for large collections. Shakespeare's works → 32,000 terms × ~37 plays = manageable. But 1,000,000 web docs × 500,000 terms = 500 billion cells.

Solution: Store only the 1s — the inverted index.

1.3 The Inverted Index

An inverted index consists of:

  • Dictionary: a sorted list of all distinct terms (vocabulary).
  • Postings lists: for each term, a list of the documents (docIDs) in which it appears.

Building a Basic Inverted Index

  1. Make a pass through the collection assembling all term–docID pairs.
  2. Sort the pairs with the term as dominant key and docID as secondary key.
  3. Organize docIDs for each term into a postings list; compute term and document frequency statistics.
Example: For "friends romans countrymen" — after tokenization and normalization: friend → [doc1, doc2], roman → [doc2], countryman → [doc1].

1.4 Processing Boolean Queries

Boolean queries use AND, OR, NOT operators. Processing uses postings list operations.

AND Query (Intersection)

Walk through both postings lists simultaneously in sorted order. If docIDs match → add to result. Otherwise advance the pointer with the smaller docID.

Complexity: O(m + n) where m = |postings(t1)|, n = |postings(t2)|

Query Optimization for AND

Key principle: Process terms in increasing order of document frequency. Start with the term that has the fewest postings — it will prune the result set fastest. This minimizes the number of comparisons.

NOT Operator

Use a bit-vector or complement. "NOT X" = all docIDs not in postings(X). OR: walk two lists and exclude matches.

OR Query (Union)

Walk both lists; add docID if it appears in either. Also O(m + n).

1.5 Extended Boolean Model vs. Ranked Retrieval

Limitations of the Boolean model:

  • Requires users to formulate precise Boolean expressions — most users find this unnatural.
  • Binary relevance: a document either matches or does not — no notion of "how well" it matches.
  • AND queries often return too few results; OR queries return too many.

Extensions include proximity search (e.g., WestLaw's /s = within sentence, /n = within n words) and ranked retrieval (introduced in Chapter 6).

Be ready to: (1) draw the term-document matrix for a small collection, (2) process a Boolean AND/OR/NOT query step-by-step using postings lists, (3) explain why query optimization (ordering terms by frequency) is important.
Chapter 2

The Term Vocabulary and Postings Lists

2.1 Key Definitions

TermDefinition
WordA delimited string of characters as it appears in the text.
TermA "normalized" word (case, morphology, spelling, etc.); an equivalence class of words — the actual entry in the IR dictionary.
TokenAn instance of a word or term occurring in a document (after splitting).
TypeAn equivalence class of tokens; generally the same as "term."

2.2 Determining the Vocabulary of Terms

2.2.1 Tokenization

Tokenization splits a stream of characters into tokens. Issues include:

  • Apostrophes: Finland's → "Finland" and "s"? Or "Finlands"?
  • Hyphens: Hewlett-Packard → one or two tokens? co-education?
  • Numbers: 3/20/91, B-52, (800) 234-2333
  • Multi-word entities: "San Francisco" — one token or two?
  • Languages: French (elision), German (compound words), Chinese/Japanese (no spaces), Arabic (right-to-left)

2.2.2 Stop Words

Stop words are the most common words that carry little semantic content: the, a, and, to, be. The top 30 words account for ~30% of all tokens.

Arguments FOR stop lists
  • Reduces index size significantly.
  • Common words have little discriminative power for retrieval.
Arguments AGAINST stop lists
  • Good compression makes storage cheap.
  • Needed for phrase queries: "King of Denmark", "To be or not to be"
  • Modern query optimization minimizes query-time cost.

Current trend: Away from using stop lists, especially in large-scale systems.

2.2.3 Normalization

Normalize indexed text and query words into the same form. Common techniques:

  • Case folding: Reduce all letters to lower case. Exception: mid-sentence proper nouns (General Motors).
  • Accents/diacritics: Normalize résumé→resume, Tübingen→Tubingen (users often omit accents).
  • Equivalence classes: car = automobile (via thesaurus), colour = color.
  • Asymmetric expansion: "window" → search for "window, windows"; "Windows" → search only "Windows".

2.2.4 Stemming

A rule-based process that reduces different forms of words to a stem (root). Addresses dimensionality reduction.

Porter Stemmer (most widely used): Initialization (lowercase, keep only letters/digits) → 1 prefix phase → 5 suffix-stripping phases.
Example phases: Step 1 removes -s, -es, -ied; Step 2 replaces suffixes like -tional→-tion; Steps 3–5 remove more complex suffixes.

Effect on a 10,000-word vocabulary: Reduces vocabulary by ~1/3 (6,370 distinct stems).

Trade-off: Stemming increases recall but can hurt precision (operative, operational, operating all → "oper").

2.2.5 Lemmatization

Reduces inflectional/variant forms to proper dictionary headword (lemma) using morphological analysis and context.

Stemming
  • Rule-based, fast, no context needed.
  • May produce non-words ("oper").
  • Heuristic — can over-stem or under-stem.
  • Better recall, sometimes worse precision.
Lemmatization
  • Uses dictionary + context (POS tagging).
  • Produces valid dictionary forms.
  • Slower and harder to implement.
  • Better precision, but recall gains smaller.

2.2.6 Morphology

  • Inflectional morphology: Suffix added without changing grammatical category — run → runs, running, ran. Semantically regular.
  • Derivational morphology: Creates new words, may change category — nation (noun) → national (adj) → nationalize (verb). Semantically irregular.
⚑ Exam Tip
Classic exam question: "Are these statements True/False? (1) Stemming never lowers precision — FALSE. (2) Stemming never lowers recall — TRUE. (3) Stemming increases vocabulary size — FALSE. (4) Stemming should only be applied at indexing time — FALSE (must apply to queries too)."

2.3 Skip Pointers for Faster Postings Intersection

Standard AND merge is O(m+n). Skip pointers allow jumping ahead in a postings list, avoiding comparisons with entries that will not be in the intersection.

How Skip Pointers Work

At each skip point, we store a pointer ahead in the list and the docID at that position. If docID(skip(p1)) ≤ docID(p2), we can skip p1 ahead to skip(p1) without visiting intermediate entries.

Where to Place Skip Pointers

Simple heuristic (Moffat & Zobel, 1996): For a postings list of length L, place √L evenly spaced skip pointers.

Trade-off: More skips → shorter skip spans → more pointer comparisons. Fewer skips → fewer comparisons but rarely able to skip. The √L rule balances these.

Note: With modern memory-based indexing, the I/O savings from skip pointers may be less significant than the overhead of extra comparisons.

2.4 Positional Postings and Phrase Queries

2.4.1 Bi-word Indexes

Index every consecutive pair of terms as a phrase. "Friends Romans Countrymen" → "friends romans" + "romans countrymen".

  • Pro: Two-word phrase queries are immediate.
  • Con: Longer phrases require Boolean combination of biwords, which can produce false positives. Dictionary bloat (doubles index size).

Extended biwords: Use POS tagging to identify N X*N patterns (Noun-Article/Prep-Noun) as extended biwords. Example: "coins are in pocket" → NX XN → index "coin pocket".

2.4.2 Positional Indexes

Store for each term the positions in which it appears within each document:

⟨term, doc_freq; doc1: pos1, pos2, …; doc2: pos1, …; …⟩

For phrase queries, use a merge algorithm that checks positional proximity. For "to be or not to be", we need positions where all terms appear in consecutive order.

Positional Intersect Algorithm

For a k-word phrase: merge docID lists first; then for matching docIDs, check that positions form a consecutive sequence (positions differ by exactly 1 for each successive term pair).

For proximity queries (WITHIN k words): check |pos(t1) - pos(t2)| ≤ k.

Positional Index Size

Document sizeRegular postingsPositional postings
1,000 terms (web page)1 per doc1 per doc
100,000 terms (book)1 per doc100 per doc

Rules of thumb: Positional index is 2–4× larger than non-positional index; takes 35–50% of original text volume.

Combination Strategy

Use biword index for common phrases and positional index for general phrase/proximity queries. Williams et al. (2004): a mixed scheme executes a typical web query mix in ¼ of the time of positional-only, at a 26% space overhead.

⚑ Exam Tip
Understand why biword indexes have false positives and why positional indexes are larger. Be able to trace through a positional intersect algorithm for a 2–3 term phrase query.
Chapter 3

Dictionaries and Tolerant Retrieval

3.1 Search Structures for Dictionaries

Two main data structures for the dictionary:

Hash Tables
  • O(1) lookup — very fast.
  • No prefix search (can't find all words starting with "hyp").
  • Can't find minor variants easily.
  • Expensive rehashing when vocabulary grows.
  • Not suitable for wildcard or spelling-correction lookups.
B-Trees
  • O(log M) lookup — slightly slower.
  • Supports prefix search natively.
  • Supports range queries on ordered values.
  • Balancing adds complexity (B-trees mitigate this vs binary trees).
  • Preferred for tolerant retrieval scenarios.

3.2 Wildcard Queries

Users use wildcard queries when uncertain of spelling, aware of multiple spellings, or dealing with foreign words. The * matches any sequence of characters.

Trailing Wildcard: mon*

Easy with B-tree: retrieve all terms in range mon ≤ w < moo.

Leading Wildcard: *mon

Maintain a second reversed B-tree. Reverse both the query and dictionary terms: look up range nom ≤ w < non in the reversed tree.

General Wildcard: co*tion

Problem: the wildcard is in the middle. Two solutions:

3.2.1 Permuterm Index

For each term, add a $ sentinel and generate all rotations:

hello → hello$, ello$h, llo$he, lo$hel, o$hell

Index all rotations; each rotation's key points to the original term.

Query lookup table:

Query PatternLookup Key
X*$X*
*XX$*
*X*X*
X*YY$X*
Example: hel*o (X=hel, Y=o) → look up o$hel* in permuterm index.

Drawback: Empirically quadruples the lexicon size for English.

3.2.2 Bigram (k-gram) Indexes

Enumerate all k-grams (k=2 is most common) in every term, including a special $ word-boundary symbol. Build a second index: bigram → list of vocabulary terms containing it.

Query: For mon*: compute bigrams $m, mo, on → retrieve all terms matching these bigrams → post-filter against original pattern mon*.

Advantage over Permuterm: Space-efficient (no quadrupling). Allows overlap Jaccard-style scoring. Existing inverted index infrastructure can be reused.

Drawback: Requires post-filtering because the bigram index is a superset (false positives may appear). Query execution may result in large disjunctions.

3.3 Spelling Correction

Two Uses

  • Document correction: Especially for OCR'd documents. Domain-specific errors (rn/m confusion).
  • Query correction: "Did you mean X?" suggestions for misspelled queries.

Two Flavors

  • Isolated-word correction: Check each word independently. Will miss context errors like "from" → "form" (both valid words).
  • Context-sensitive correction: Use surrounding context to detect errors. Needed for "I flew form Heathrow" → should be "from".

3.3.1 Edit Distance (Levenshtein Distance)

Definition

The minimum number of single-character edit operations (insert, delete, replace, optionally transpose) needed to transform one string into another.

Examples: dof→dog = 1 (replace f→g). cat→act = 2 (or 1 with transposition). cat→dog = 3.

EditDistance(s₁, s₂) — Dynamic Programming
Initialize m[i,j] = 0
for i = 1 to |s₁|: m[i,0] = i
for j = 1 to |s₂|: m[0,j] = j
for i = 1 to |s₁|:
  for j = 1 to |s₂|:
   m[i,j] = min(
    m[i-1,j-1] + (s₁[i]≠s₂[j] ? 1 : 0), // replace
    m[i-1,j] + 1, // delete
    m[i,j-1] + 1 // insert
   )
return m[|s₁|][|s₂|]

Using Edit Distance for Spelling Correction

Given a query term, enumerate all character sequences within a preset edit distance (e.g., 2), intersect with the valid dictionary, and suggest the most frequent matches.

Alternative: Run the query and if few results are returned, try spelling-corrected versions.

3.3.2 n-gram Overlap for Spelling Correction

Break both the query and dictionary terms into n-grams (usually bigrams). Use the bigram index to retrieve all dictionary terms sharing bigrams with the query. Threshold by number of overlapping bigrams.

Example: Query "lord" → bigrams: lo, or, rd. Retrieve all dictionary terms with ≥2 matching bigrams. Post-score by Jaccard or overlap coefficient.

Context-Sensitive Spelling Correction

For phrase queries: if "flew form Heathrow" returns no results, generate alternative spellings for each word and try all combinations. Use hit-based correction: suggest the alternative with the most document matches.

3.4 Phonetic Correction (Soundex)

Soundex groups phonetically similar words (especially names) under the same code. Invented in 1918 for the U.S. census.

Soundex Algorithm

  1. Retain the first letter of the word.
  2. Change all A, E, I, O, U, H, W, Y to '0' (zero).
  3. Replace remaining letters with digits:
    • B, F, P, V → 1
    • C, G, J, K, Q, S, X, Z → 2
    • D, T → 3
    • L → 4
    • M, N → 5
    • R → 6
  4. Remove all pairs of consecutive identical digits.
  5. Remove all zeros.
  6. Pad with trailing zeros; return first 4 characters (letter + 3 digits).

Example: Herman → H655. Mary → M600. Nira → N600 (same code → phonetically equivalent).

Limitation: Soundex is provided by most databases (Oracle, Microsoft) but performs poorly for IR compared to specialized phonetic algorithms (Zobel & Dart, 1996). It fails for Chebyshev/Tchebycheff (different codes despite similar pronunciation).
⚑ Exam Tip
Be ready to compute Soundex codes from scratch. Also understand when to use each wildcard technique (Permuterm vs. k-gram) and the trade-offs.

Index Construction

4.1 Hardware Basics

Many design decisions in IR are driven by hardware characteristics.

ParameterValue (circa 2007)
Average disk seek time (s)5 ms = 5 × 10⁻³ s
Transfer time per byte (b)0.02 µs = 2 × 10⁻⁸ s
Processor clock rate10⁹ s⁻¹
Low-level operation (p)0.01 µs = 10⁻⁸ s
Main memorySeveral GB
Disk space1 TB or more

Key Hardware Principles

  • Memory >> Disk speed: Access to data in memory is orders of magnitude faster than disk.
  • No transfer during seeks: While the disk head is being positioned, no data is transferred. Minimize seeks by reading large contiguous chunks.
  • Block-based I/O: Disk reads/writes are in blocks (8 KB–256 KB). Use this to your advantage.
  • Fault tolerance: Using many cheap commodity machines (distributed) is far cheaper than one fault-tolerant machine.

4.2 Blocked Sort-Based Indexing (BSBI)

Used when the entire collection does not fit in memory. Segments the collection into fixed-size blocks, sorts each in memory, and then merges.

BSBI Algorithm

BSBINDEXCONSTRUCTION()
n ← 0
while (all documents have not been processed):
  n ← n + 1
  block ← PARSENEXTBLOCK()
  BSBI-INVERT(block) // sort termID-docID pairs, build index
  WRITEBLOCKTODISK(block, fₙ)
MERGEBLOCKS(f₁, …, fₙ; f_merged)

Steps

  1. Segment: Divide collection into equal-size blocks (that fit in memory).
  2. Sort: For each block, sort termID–docID pairs in memory.
  3. Write: Store sorted intermediate results on disk.
  4. Merge: Merge all intermediate results into the final index.
Time complexity: Θ(T log T) where T is the total number of termID–docID pairs. The dominant cost is sorting. The actual bottleneck is usually parsing (PARSENEXTBLOCK) and final merge (MERGEBLOCKS).

Merging in BSBI

Two blocks are loaded into memory, merged (like merge-sort's merge step), and written back to disk. For n blocks, this is done iteratively or in a multi-way merge.

4.3 Single-Pass In-Memory Indexing (SPIMI)

SPIMI avoids the need for a global termID mapping by generating separate dictionaries for each block.

Two Key Ideas

  • Key idea 1: Generate a separate dictionary for each block — no need to maintain a term↔termID mapping across blocks.
  • Key idea 2: Don't sort. Accumulate postings directly in postings lists as they occur (using a hash map for the dictionary).
SPIMI-INVERT(token_stream)
output_file = NEWFILE()
dictionary = NEWHASH()
while (free memory available):
  token ← next(token_stream)
  if term(token) ∉ dictionary:
   postings_list = ADDTODICTIONARY(dictionary, term(token))
  else:
   postings_list = GETPOSTINGSLIST(dictionary, term(token))
  if full(postings_list):
   postings_list = DOUBLEPOSTINGSLIST(dictionary, term(token))
  ADDTOPOSTINGSLIST(postings_list, docID(token))
sorted_terms = SORTTERMS(dictionary)
WRITEBLOCKTODISK(sorted_terms, dictionary, output_file)
return output_file

4.4 Distributed Indexing (MapReduce)

Web-scale search engines cannot build an index on a single machine. Distributed indexing is the solution.

Architecture

  • Master machine: Directs the indexing job; considered "safe".
  • Parsers: Assigned document splits by master; emit (term, docID) pairs; partition into j segment files by term.
  • Inverters: Collect all (term, docID) pairs for one term-partition; sort and write to postings lists.

MapReduce Framework

PhaseDescription
Mapinput (web collection) → list(termID, docID). Each parser processes documents and emits (term, doc) pairs.
Reduce(termID₁, list(docID)), (termID₂, list(docID)), … → postings lists. Inverters sort and write final postings.
Scale: Google circa 2007: ~1 million servers, 3 million processors/cores (Gartner estimate). Data centers distributed globally. Uses commodity machines instead of expensive fault-tolerant hardware. Original MapReduce publication: Dean and Ghemawat (2004).

4.5 Dynamic Indexing

Real collections are not static — documents arrive, are deleted, and modified over time.

Simple Approach (Auxiliary Index)

  • Maintain a large main index on disk.
  • New documents go into a small auxiliary index in memory.
  • Search across both and merge results.
  • For deletions: use an invalidation bit-vector; filter output by this vector.
  • Periodically re-index everything into one main index.

Google Dance

Search engines do incremental indexing (auxiliary) continuously and periodically do full background re-indexing ("Google Dance"). This keeps the index fresh while ensuring correctness.

Chapter 5 (Sections 5.1–5.3)

Index Compression

Why Compression?

  • Less disk space → saves money.
  • More in memory cache → faster retrieval (fewer disk seeks).
  • Faster disk-to-memory transfer: Reading compressed data + decompressing is often faster than reading uncompressed data. Decompression algorithms are very fast on modern hardware.
Key insight: In most cases, the retrieval system runs faster on compressed postings lists than on uncompressed ones, because the time savings from smaller I/O outweigh the decompression cost.

Lossless vs. Lossy compression: The techniques in this chapter are lossless. Stemming, stop-word elimination, and case folding are forms of lossy compression.

5.1 Statistical Properties of Terms

5.1.1 Heaps' Law — Vocabulary Size

Heaps' Law

Vocabulary size M as a function of collection size T (number of tokens):

M = k · T^b Typical values: 30 ≤ k ≤ 100, b ≈ 0.5 For Reuters-RCV1 (best fit): k = 44, b = 0.49 → M = 44 × T^0.49

Interpretation:

  • Dictionary size continues to grow with more documents — no maximum vocabulary is reached.
  • The dictionary is quite large for large collections.
  • Factors that increase M: including numbers, spelling errors.
  • Factors that decrease M: case folding, stemming.
Example: For RCV1, first 1,000,020 tokens → Heaps predicts 38,323 terms. Actual: 38,365. Very close!

5.1.2 Zipf's Law — Term Distribution

Zipf's Law

The collection frequency cf_i of the i-th most common term is proportional to 1/i:

cf_i ∝ 1/i → cf_i = c/i Equivalently: log(cf_i) = log(c) − log(i) This is a power law with exponent k = −1. Rank × Frequency ≈ constant

Consequences: If the most frequent term ("the") appears cf₁ times, then the 2nd most frequent ("of") appears cf₁/2 times, the 3rd ("and") appears cf₁/3 times, etc.

Practical implication for compression: Most terms are rare (long tail). Frequent terms have very small gaps in their postings lists; rare terms have large gaps. This non-uniform distribution motivates variable-length encoding.

5.2 Dictionary Compression

Reuters-RCV1 Baseline

400,000 terms. Naïve fixed-width storage: 20 bytes/term + 4 bytes/freq + 4 bytes/postings pointer = 28 bytes × 400,000 = 11.2 MB.

Problems: average English term is ~8 characters, so 20-byte slots waste space. Terms over 20 chars can't be stored.

5.2.1 Dictionary as a String

Store all terms as one long concatenated string. Replace fixed-width term storage with variable-length term storage + a term pointer.

  • Total string length = 400,000 × 8 bytes = 3.2 MB.
  • Pointers need to resolve 3.2M positions → log₂(3.2M) ≈ 22 bits = 3 bytes per pointer.
  • New cost: 4 (freq) + 4 (postings ptr) + 3 (term ptr) + 8 (avg term) = 19 bytes/term → 400,000 × 19 = 7.6 MB.

5.2.2 Blocked Storage

Group terms into blocks of size k. Store a term pointer only for the first term of each block. Prepend each term with a 1-byte length indicator.

  • For k=4: save (k−1)×3 = 9 bytes per block in pointers, add k=4 bytes for lengths. Net saving: 5 bytes per 4-term block.
  • Total savings: 400,000 × (1/4) × 5 = 0.5 MB → down to 7.1 MB.
  • Trade-off: Binary search finds the block; then linear scan within the block. Slower lookup for larger k.

Front Coding

Exploit common prefixes in alphabetically sorted terms. Mark end of common prefix with *, replace prefix in subsequent terms.

8automat*a → 8 automat a (automate) 8automate → 1 e 9automatic → 2 ic 10automation → 3 ion

Front coding saves about 1.2 MB more for Reuters-RCV1 → 5.9 MB.

5.3 Postings File Compression

Postings file is much larger than dictionary. Reuters-RCV1: 100M postings, 20 bits each = 250 MB uncompressed (or 400 MB as 32-bit integers).

Gap Encoding (d-gaps)

Instead of storing absolute docIDs, store the differences (gaps) between consecutive docIDs. Frequent terms have small gaps (mostly 1); rare terms have gaps similar in size to the docIDs themselves.

Example for "the": docIDs: 283042, 283043, 283044, 283045... gaps: 283042, 1, 1, 1... Example for "arachnocentric": gaps are as large as docIDs (~20 bits)

Goal: encode small gaps with fewer bits. Use variable-length encoding.

5.3.1 Variable Byte (VB) Encoding

Encode gaps using an integral number of bytes. Each byte has:

  • 7 payload bits (from the low-order end).
  • 1 continuation bit (high-order): 1 = last byte of this number, 0 = more bytes follow.
VB encoding examples: Gap = 5 → 1000 0101 (single byte, continuation=1, value=5) Gap = 824 → 0000 0110 1011 1000 (two bytes: 6 × 128 + 56 = 824) Gap = 214577 → three bytes
VBEncodeNumber(n)
bytes ← []
while true:
  Prepend(bytes, n mod 128) // add 7-bit chunk
  if n < 128: break
  n ← n div 128
bytes[last] += 128 // set continuation bit on last byte
return bytes
Result for Reuters-RCV1: VB encoding → 116 MB (from 250 MB). >50% reduction. Simple to implement and fast to decompress.

5.3.2 Gamma (γ) Encoding

A bit-level code that is within a factor of 2 of optimal for any probability distribution. Encodes a gap G as: length (in unary) + offset (binary without leading 1).

For gap G: offset = G in binary with leading 1 removed length = length of offset, encoded in unary (n 1s followed by 0) γ code(G) = length + offset Example: G = 13 = 1101₂ offset = 101 (removed leading 1) length = 3 bits → unary = 1110 γ code = 1110 101

Chapter 6 (Sections 6.2–6.3)

Scoring, Term Weighting, and the Vector Space Model

Why Move Beyond Boolean?

Boolean retrieval returns a set of documents that either match or don't. For large collections, the matching set is often too large to be useful. We need to rank matching documents by how well they match the query.

This requires assigning a numerical score to each (query, document) pair — the score reflects relevance or match quality.

6.2 Term Frequency and Weighting

Bag-of-Words Model

In the vector space model, we ignore the order of terms and treat a document as a "bag" of words. A document is represented by the (weighted) frequencies of each term it contains.

Term Frequency (tf)

Definition

tf_{t,d} = the number of times term t occurs in document d.

Raw tf alone is problematic: a term appearing 10 times in a document is not 10× more relevant than one appearing once. But it should count for more than once. Standard fix: use the log-frequency weight.

Log-frequency weight: w_{t,d} = 1 + log₁₀(tf_{t,d}) if tf_{t,d} > 0 w_{t,d} = 0 if tf_{t,d} = 0

Document Frequency (df) and Inverse Document Frequency (idf)

Definition

df_t = the number of documents in which term t appears. idf_t = log₁₀(N / df_t), where N is the total number of documents.

Intuition: Rare terms are more informative (discriminative) than frequent terms. A term that appears in every document tells us nothing about which document is relevant. idf gives high weight to rare terms and low weight to frequent ones.

idf_t = log₁₀(N / df_t) Examples (N = 1,000,000 docs): "the" → df ≈ 1,000,000 → idf = log₁₀(1) = 0 (tells us nothing) "car" → df ≈ 10,000 → idf = log₁₀(100) = 2 (moderately informative) "gnat" → df ≈ 100 → idf = log₁₀(10,000) = 4 (very informative)

tf-idf Weight

The tf-idf weight of term t in document d is the product of its log-tf and idf:

tf-idf_{t,d} = (1 + log₁₀(tf_{t,d})) × log₁₀(N / df_t) This is the most widely used weighting scheme in IR. High tf-idf means: term occurs frequently in this document (high tf) AND rarely in the collection (high idf).

Scoring a Query

For a free-text query q (treated as a set of terms), the score of document d for query q is:

Score(q, d) = Σ_{t ∈ q ∩ d} tf-idf_{t,d}

Only terms appearing in both the query and the document contribute. Terms with zero tf or zero idf contribute nothing.

⚑ Exam Tip
Watch out: idf of a term appearing in all N documents = log(N/N) = log(1) = 0. Such terms get zero weight. This makes intuitive sense — if every document contains a term, it doesn't help distinguish between them.

6.3 The Vector Space Model for Scoring

Document as Vector

Each document d is represented as a vector in a |V|-dimensional space, where V is the vocabulary. The weight of term t in document d is its tf-idf weight:

d = (w_{t₁,d}, w_{t₂,d}, …, w_{t|V|,d})

Most dimensions are 0 (terms not in document). The vector space is extremely high-dimensional and sparse.

Query as Vector

The query q is also represented as a vector in the same space. For a query term t, its weight can be computed similarly (e.g., tf in the query × idf of t).

Cosine Similarity

To score a document d against query q, we measure the cosine of the angle between their vectors. This normalizes for document length.

cos(q, d) = (q · d) / (|q| × |d|) = Σ_{t} (w_{t,q} × w_{t,d}) / (sqrt(Σ_t w²_{t,q}) × sqrt(Σ_t w²_{t,d}))
Why cosine? Two identical documents will have cosine = 1 regardless of length. A document that is twice as long but contains the same terms in similar proportions should score just as high — cosine normalization handles this. Range: [0, 1] for non-negative weights.

Unit Vectors (Length Normalization)

Computing cosine is equivalent to dot product after normalizing both vectors to unit length:

d̂ = d / |d| (normalize to unit vector) cosine similarity = q̂ · d̂ = Σ_t (q̂_t × d̂_t)

The SMART Notation for tf-idf Variants

Systems use different combinations of tf and idf weighting. The SMART notation encodes these as three components: tf-idf-normalization.

ComponentSymbolFormula

l (log)1 + log(tf_{t,d})
a (augmented)0.5 + 0.5 × tf / max_tf
b (boolean)1 if tf > 0, else 0

A common scheme: lnc.ltc = log-tf, no idf, cosine norm for docs; log-tf, idf, cosine norm for queries.

⚑ Exam Tip
The core VSM exam skills are: (1) Compute tf-idf weights from raw tf and df values. (2) Represent documents as tf-idf vectors. (3) Compute cosine similarity between a query vector and a document vector, step by step. (4) Explain why length normalization matters. (5) Know why idf of a term in all documents = 0.

Advantages of VSM

  • Partial matching — documents don't need to contain all query terms.
  • Natural ranking — documents with more/rarer query terms score higher.
  • Length normalization — longer documents don't automatically score higher.
  • Theoretically grounded and empirically effective.

Limitations of VSM

  • Independence assumption: treats terms as independent (ignores synonymy, polysemy).
  • Bag-of-words: ignores word order and proximity.
  • The "right" weighting scheme is still an open empirical question.
