📘 Chapter 1 — Introduction to Software Engineering

Lecture 1b & Lecture 2 slides

What is Software Engineering?

An engineering discipline concerned with all aspects of software production — from early specification to maintaining systems after deployment. It covers technical processes, project management, and tool development.

⚡ Key Distinction
Computer Science → Theory & fundamentals  |  Software Engineering → Practical development & delivery of useful software  |  System Engineering → All computer-based systems including hardware

Software costs: ~60% development + 40% testing. For long-life systems, maintenance >> development cost.

Why Do Software Projects Fail?

  • Increasing complexity — New demands arise faster than engineering methods evolve.
  • Failure to use engineering methods — "Build it then fix it" approach leads to expensive, unreliable software.

Essential Attributes of Good Software

EfficiencyNo wasteful use of resources (memory, CPU, response time)
AcceptabilityUnderstandable, usable, and compatible for its intended users

Application Types

  • Stand-alone — Runs locally, no network needed (e.g., a desktop text editor)
  • Interactive transaction-based — Remote server, user access via browser (e.g., e-commerce sites)
  • Embedded control systems — Control hardware devices; most numerous type (e.g., car engine controller)
  • Batch processing — Process large data sets (e.g., bank statements)
  • Entertainment systems — Personal use, entertainment (e.g., games)
  • Modelling & simulation — Scientists model physical processes
  • Data collection systems — Use sensors to collect data (e.g., weather stations)

Software Engineering Ethics

Engineers must behave honestly and ethically, beyond just following laws.

  • Confidentiality — Respect employer/client confidentiality, even without a formal agreement
  • Competence — Honestly quote skill level; never accept work beyond your competence
  • Intellectual Property Rights — Respect patents, copyright; protect employer IP
  • Computer Misuse — Never misuse technical skills (e.g., spreading viruses is wrong; even trivial misuse like playing games on an employer's machine)
🔴 Ethical Dilemmas to Know
Releasing a safety-critical system without completing testing  |  Developing military/nuclear weapons systems (ethically questionable but strategically considered)  |  Disagreeing with senior management policies

Web Software Engineering

  • Software reuse — Assemble from pre-existing components (preferred approach)
  • Incremental & Agile — Build and deliver incrementally (can't specify all requirements upfront)
  • Service-oriented — Use independent web services (e.g., Google Maps, PayPal, Facebook login)
  • Rich Interfaces — AJAX, HTML5 enable rich browser-based UIs

General Issues Affecting Software

  • Heterogeneity — Systems must run on diverse networks and device types
  • Business & Social Change — Rapid change in business requires rapid software change
  • Security & Trust — Software is interwoven with daily life; must be trustworthy
  • Scale — From tiny embedded systems to global cloud platforms

📗 Chapter 2 — Software Processes

chapter_2__1_.pptx

What is a Software Process?

A structured set of activities required to develop a software system. A process model is an abstract description of a process from a specific perspective.

Every process includes: Products (outcomes), Roles (responsibilities), Pre/Post conditions (what's true before/after an activity).

Plan-Driven

All activities planned upfront. Progress measured against the plan. Suitable for stable requirements.

Agile

Planning is incremental. Adapt plans as customer requirements change. Less documentation.

The 3 Core Process Models

🌊 1. Waterfall Model (Plan-Driven)

Sequential phases — each must be complete before the next starts.

  1. Requirements analysis & definition
  2. System & software design
  3. Implementation & unit testing
  4. Integration & system testing
  5. Operation & maintenance

When to use: Large systems engineering projects; stable, well-defined requirements; multi-site development (plan-driven coordination is easy).

Drawback: Inflexible to changing requirements. Stable requirements are very rare in practice.

🔄 2. Incremental Development

Specification, development, and validation are interleaved. Deliver working software in increments.

✅ Benefits

  • Reduced cost of change (less rework)
  • Easier customer feedback (demos of actual software)
  • Rapid delivery of useful software

❌ Problems

  • Process not visible to managers (no deliverables to measure progress)
  • System structure degrades over time without refactoring
  • Poor documentation (skipped to reduce cost)
♻️ 3. Integration & Configuration (Reuse-Oriented)

Assemble systems from existing reusable/configurable components (COTS — Commercial Off-The-Shelf). Reuse is the standard approach for many business systems today.

Key stages: Requirements spec → Software discovery & evaluation → Requirements refinement → Configuration → Component adaptation & integration

✅ Advantage

Reduced cost and risk; faster delivery

❌ Disadvantage

Requirements compromises are inevitable; loss of control over evolution of reused elements

Design Activities (Process Activities → Design)

  • Architectural design — Identify overall structure, major components/subsystems
  • Database design — Design data structures and representation in DB; new or reuse existing DB?
  • Interface design — Define component interfaces; must be user-friendly and hide complexity
  • Component selection & design — Search for reusable components; if unavailable, design from scratch with UML diagrams

Software Validation (Testing)

V&V (Verification & Validation) — Show system conforms to specification and meets customer requirements.

  • Component (Unit) Testing — Individual components tested independently by developers. Tools: JUnit (Java), VS Code plugins.
  • System Testing — Test entire system after integrating modules. Tests interactions between components.
  • Customer Testing (Beta) — Test with real customer data; reveals omissions and errors in requirements.
V-Model (Improvement of Waterfall)
Each development phase has a corresponding testing phase. Plan-driven approach where testing is planned alongside development.

Alpha testing → by development staff  |  Beta testing → by real end users

Coping with Change

Change is inevitable in all large projects. Cost of change = cost of rework (re-analyzing requirements) + cost of implementing new module.

Proactive: Change Anticipation

Include activities to anticipate possible changes before significant rework.
Example: Prototyping

Reactive: Change Tolerance

Design the process so changes can be accommodated at low cost.
Example: Incremental Delivery

🔬 Software Prototyping

An initial version of a system used to demonstrate concepts and explore design options. Used to:

  • Refine requirements elicitation and validation
  • Explore design options
  • Run back-to-back tests

Important: Only prototype unclear/risky parts. Skip error handling and NFRs (response time, memory) in prototypes.

Throw-away prototypes: Must be discarded — they are undocumented, structure is degraded, won't meet quality standards, can't be tuned for NFRs.

Benefits: Improved usability, closer match to real needs, improved design quality, reduced development effort, improved maintainability.

📦 Incremental Delivery
Break development into increments; deliver each increment to the user. User prioritizes requirements — highest priority goes first. Requirements for current increment are frozen during development.

Advantages: Early increments act as prototypes; lower risk of project failure; highest-priority services get most testing.

Problems: Hard to detect dependencies between increments; difficult to form a complete contract; may cause budgetary issues.

Software Process Improvement

Goal: Deliver quality software with reduced cost and faster timelines.

Software Process Improvement = Understanding existing processes + changing them to increase quality and/or reduce costs.

Process Maturity Approach

Focus on improving process management. Introduces good engineering practices. Higher maturity = better technical & management practice. Generally for plan-driven approaches.

Agile Approach

Focus on iterative development and reducing overheads. Rapid delivery + responsiveness to change. Code-first with minimal documentation.

Process Improvement Cycle: Measure process → Analyze weakness/strengths → Change process → Measure impact → Repeat

Process Metrics: Time taken for activities | Resources required (person-days) | Number of defects discovered

SEI Capability Maturity Model (CMM) — 5 Levels

🎯 Exam Favourite — Know All 5 Levels!
CMM is NOT a software process model. It rates organizational maturity on a 5-point scale.
1
Initial Unmanaged work. No documentation. No project management plans. No cost estimation. "Build and Fix" model. Most organizations are here. Challenges: Planning, SQA.
2
Managed Project goals satisfied. Defined organizational policies (which process to use). Documented project plan. Proper resource management. Usually same type of software.
3
Defined Process management procedures defined and used. Proper teamwork and hierarchy. Collaborative learning. Works on diverse application types.
4
Quantitatively Managed Assess organizational performance with metrics. Better project management. Risk analysis and proactive approaches (what if a project fails?).
5
Optimising Process improvement strategies defined and actively used. Increased adaptation to newer trends and technologies. Innovative solutions.
📝 Case Study Examples (CMM Level ID)
  • Norton LifeLock → Level 5 (Optimising) — Innovative security solutions, proper team structure, prototyping, performance prediction
  • Everyday Wear (e-commerce app) → Level 3 (Defined) — Diverse apps (desktop + mobile), managing diverse work
  • ABC Co. (Smart Gadgets) → Level 4 (Quantitatively Managed) — Structured teams, risk assessment for modules

📙 Chapter 3 — Agile Software Development

chapter_3.pptx

Why Agile?

Business needs change rapidly → requirements are unstable → software must adapt quickly. Plan-driven approaches delay delivery and may produce obsolete software by delivery time.

Agile methods emerged in the late 1990s to radically reduce delivery time for working software.

⚡ Agile Key Characteristics
  • Specification, design, and implementation are interleaved
  • Developed as a series of versions/increments (stakeholders evaluate each)
  • Frequent delivery every 2–3 weeks
  • Extensive tool support (e.g., Selenium for automated testing)
  • Minimize documentation overhead

5 Principles of Agile Methods

PrincipleDescription
Customer InvolvementCustomers closely involved throughout. They provide/prioritize requirements and evaluate iterations.
Incremental DeliverySoftware developed in increments; customer specifies what goes in each.
People Not ProcessRecognize and exploit team skills. Let them develop their own working styles.
Embrace ChangeExpect requirements to change; design to accommodate changes.
Maintain SimplicityFocus on simplicity in software AND process. Actively eliminate complexity.

Extreme Programming (XP)

Takes an "extreme" approach to iterative development. Requirements are expressed as user stories. All tests must pass for every build.

📖 XP Concepts
  • User Stories — Simple, informal customer statements about needed functionality. Written on cards (Task Cards).
  • Metaphors — Show how the system would work, proposed by the project team based on user stories.
  • Spike — A very simple program built to explore suitability of a proposed solution (similar to prototype).

XP Activities: Listening → Designing → Coding → Testing → Feedback

Best for: Small-scale projects; projects using new/research technology

XP Practices — Part A

PracticeDescription
Incremental PlanningRequirements on story cards. Stories prioritized, then broken into development tasks.
Small ReleasesMinimal useful functionality first; frequent incremental releases.
Simple DesignOnly enough design to meet current requirements — no over-engineering.
Test-First DevelopmentWrite automated unit tests BEFORE implementing functionality.
RefactoringContinuously improve code as soon as improvements are found. Keep it simple and maintainable.

XP Practices — Part B

PracticeDescription
Pair ProgrammingTwo developers work at one computer. Informal review as each line checked by 2 people. Spreads knowledge, reduces risk.
Collective OwnershipAll developers work on all parts of the system. Anyone can change anything.
Continuous IntegrationAs soon as a task is done, integrate into the whole system. All unit tests must pass after integration.
On-Site CustomerCustomer representative available full-time with the XP team. Responsible for bringing requirements and participating in acceptance tests.

Test-First Development (XP's Key Innovation)

Write test cases before writing the implementation code.

Plan-Driven Order

Specification → Implementation → Testing

XP Order

Test Cases → Implementation (guided by tests) → Requirements validation

XP Testing Advantages:

  • Reduces requirements misunderstanding (tests define expected behavior)
  • Avoids test-lag (developer pace matches tester pace)
  • Customer involvement in test development
  • Tests are programs (automated) — run whenever new functionality is added
⚠️ Problems with Test-First Development
  • Programmers prefer coding → may write incomplete tests or skip edge cases
  • Some tests are hard to write incrementally (complex UI display logic)
  • Difficult to judge completeness of test set

Refactoring (XP Concept)

XP does NOT follow "design for change" (anticipate future changes upfront) — requirements may change so drastically that anticipated design becomes useless.

Instead, XP proposes constant code improvement (refactoring) — improve code even when there's no immediate need.

Examples of Refactoring
  • Reorganizing class hierarchy to remove duplicate code
  • Renaming attributes and methods to improve clarity
  • Replacing inline code with calls to library methods

Note: When deadlines are hard and implementation is critical, refactoring may be delayed.

Pair Programming

  • Two programmers sit at one computer
  • Pairs created dynamically — all members work with each other
  • Serves as an informal review process (each line seen by 2 people)
  • Spreads knowledge → reduces risk when team members leave
  • Evidence suggests a pair is more efficient than 2 programmers working separately
  • Common practice: Senior + Junior developer pairing for collaborative learning

Scrum (Agile Project Management Framework)

A framework providing structure for agile project management. Three phases:

1
Outline Planning PhaseEstablish general objectives and design software architecture.
2
Sprint CyclesSeries of sprints (2–4 weeks each), each developing an increment of the system.
3
Project Closure PhaseComplete required documentation (user manuals, help frames), assess lessons learned.

Scrum Terminology

Sprint2–4 week development iteration.
Product BacklogTo-do list for the team (user stories, tasks, architecture definitions).
Product OwnerIdentifies and prioritizes requirements. Could be customer, product manager, or stakeholder rep.
Scrum MasterGuides team, arranges daily meetings, tracks backlog, protects team from distractions. Similar to project manager.
Development TeamSelf-organizing group, max 7 people. Responsible for software and project documents.
Potentially Shippable IncrementSoftware delivered from a sprint; finished state, ready to add to final product.
Scrum (Daily Meeting)Short daily meeting reviewing progress and planning the day's work.
Sprint VelocityEstimate of product backlog effort a team can cover in one sprint. Helps in planning.
📅 Scrum Sprint Cycle (How it works)
  1. Start from Product Backlog
  2. Team selects features for the sprint
  3. Team organizes themselves to develop the sprint software
  4. During sprint: team is isolated from customer — all communication via Scrum Master
  5. At end of sprint: work reviewed and presented to stakeholders
  6. Next sprint cycle begins
Scrum Benefits
  • Product broken into manageable chunks
  • Full team visibility → improved communication
  • Customers see on-time increment delivery → feedback loop
  • Trust between customers and developers built over time

Scaling Agile Methods

Scaling Up

Using agile for large software systems that can't be developed by one small team.

Scaling Out

Introducing agile across a large organization with years of existing SW experience.

Agile works best for: Small to medium projects with co-located teams.

⚠️ Practical Problems with Agile
  • Contract issues — Most contracts need a fixed specification, but agile interleaves spec and development. Solution: Pay for developer time rather than functionality.
  • Maintenance issues — Lack of documentation makes maintenance hard. Need to keep customers involved and maintain team continuity.
  • Large systems — Distributed teams, diverse stakeholders, regulatory constraints, procurement complexity.
Multi-Team Scrum (For Large Systems)
  • Role Replication — Each team has own Product Owner and Scrum Master
  • Product Architects — Each team has an architect; they collaborate on overall architecture
  • Release Alignment — All teams align release dates for complete system demo
  • Scrum of Scrums — Daily meeting where representatives from all teams meet to discuss progress

📕 Chapter 4 — Requirements Engineering

chapter_4.pptx

What are Requirements?

Requirements = the services a customer needs from a system + constraints under which it operates.

Requirements Engineering = the process of finding out, analyzing, documenting, and checking constraints on the system. Used to create project bids, contracts.

Types of Requirements

User Requirements

  • Written in natural language (non-technical)
  • High-level view of services and constraints
  • Audience: Stakeholders, managers, non-technical end users
  • Example: "A user shall be able to search doctor appointments"

System Requirements

  • Structured document with detailed descriptions
  • Functions, services, and operational constraints
  • Defines what to implement; may form part of a contract
  • Example: "3.1 The system shall check username and password and login the system"

Functional vs Non-Functional Requirements

Functional Requirements (FR)

What the system must do — specific functions and behaviors.

Example (Hospital System):

  • Sign up / Sign in for employees/doctors/patients
  • Unique 8-digit User ID for each person
  • Search appointments by doctor
  • Generate list of expected patients per day
  • Search specialists by domain
  • View patient history, diagnosis, medications

Non-Functional Requirements (NFR)

Constraints on services or functions — system qualities, not specific behaviors.

Example (Hospital System):

  • Product: System available Mon–Fri 08:30–17:30; downtime < 5 sec/day
  • Organizational: Users authenticate with ID cards
  • External: System must respect patient privacy per code of ethics
🔗 Requirements Dependency
Requirements are not independent! One requirement often constrains another.
Example: Security (NFR) → Creates a functional requirement for user authentication. An organizational NFR (ID card auth) may lead to another NFR (card reader hardware) or alternative FR (SMS/email auth).

NFR Classification

  • Product Requirements — Constraints on runtime behavior (availability, performance)
  • Organizational Requirements — From policies/procedures of customer or developer org (platform, programming language to use)
  • External Requirements — From regulatory bodies; must be met for software approval; no data leakage or ethical misuse

Stakeholders

Any person/organization with a legitimate interest in the system who is directly affected by it.

Hospital System Stakeholders
Patients | Doctors | Nurses | Medical Receptionists | IT Staff | Healthcare Managers

Types: End users | System managers | System owners | External stakeholders (regulatory bodies)

Requirements Engineering Process

1
Requirements FeasibilityIs the system feasible given current technology and budget?
2
Requirements Elicitation & AnalysisGather requirements from stakeholders using interviews, ethnography, scenarios.
3
Requirements SpecificationDocument user and system requirements formally.
4
Requirements ValidationCheck that requirements define the system the customer actually wants.

In practice, RE is iterative — these processes are interleaved (shown as a spiral).

Requirements Elicitation Techniques

🗣️ Interviewing

Formal or informal interviews with stakeholders.

  • Closed interviews — Pre-determined questions
  • Open interviews — No fixed questionnaire; explore and refine requirements
  • Best: Mix of both types

Tips: Be open-minded; prompt users to talk by suggesting requirements; listen actively.

Problem: Communication gap; domain knowledge prerequisite needed. E.g., building e-commerce site must include transaction processing knowledge.

👁️ Ethnography

A person observes and analyzes how people actually work (rather than how they say they work).

  • People don't have to explain; observer captures real behavior
  • Social and organizational factors observed
  • Work is often richer and more complex than simple system models suggest

Scope: Requirements derived from actual work; good for fine-tuning design. But cannot suggest innovation to a system.

Focused Ethnography: Combines ethnography with prototyping. Prototype raises unanswered questions → focuses ethnographic analysis. Issue: Studies existing practices which may be historically outdated.

📖 Stories & Scenarios
Real-life examples of how a system can be used for a task. Stakeholders can relate and comment because they're practical.

Scenario structure: Starting situation → Normal flow of events → What can go wrong → Concurrent activities → Finished state

Common Problems in Requirements Elicitation

  • Stakeholders don't know what they really want
  • Stakeholders express requirements in their own terms (domain jargon)
  • Conflicting requirements from different stakeholders (e.g., HR wants employee age captured; Data Privacy team says no)
  • Organizational constraints influence requirements
  • Requirements change during the analysis process
  • New stakeholders emerge; business environment changes

Requirements Specification

Process of writing user and system requirements in a requirements document.

  • User requirements → natural language
  • System requirements → natural language + graphical notations + mathematical models
  • Use tables, forms, diagrams
🔗 Requirements and Design
Requirements = what the system should do. Design = how it does it. In practice, they are inseparable — system architecture may structure requirements; subsystems may generate new requirements; NFRs may dictate design choices (e.g., use Python for financial calculations to ensure reliability).
Problems with Natural Language
Ambiguous requirements lead to misunderstanding, system delay, and increased costs.
Bad example: "A user shall be able to search the appointments."
→ User means: search across ALL clinics  |  Developer interprets: choose clinic first, then search

Ways to Specify Requirements

  • Natural Language — Simple but ambiguous
  • Structured Specifications — Forms with defined fields (function description, inputs, outputs, pre/post conditions)
  • Tabular Specification — Supplement natural language; useful for multiple conditions/actions
  • Use Cases (UML) — Identify actors and interactions; high-level graphical model
Example: Tabular Specification (Login Module)
ConditionAction
Username correct, password wrongLogin fail
Username wrong, password wrongLogin fail
Username correct, password correctLogin success

Software Requirements Document (SRD)

The official statement of what the system developers must build. NOT a design document — it defines what, not how.

Requirements Validation

Checking that requirements define the system the customer actually wants. Overlaps with elicitation and analysis. Critical step because fixing errors later is extremely expensive.

💰 Why Validation Matters
Fixing a requirements error AFTER delivery can cost up to 100× more than fixing an implementation error. A requirements change leads to changes in design, implementation, AND testing.

Validation techniques:

  • Requirements Review — Customer and developers read the whole SRD, check for errors, anomalies, inconsistencies, then negotiate fixes.
  • Prototyping — Build a quick prototype to validate understanding.
  • Test-Case Generation — Develop test cases from requirements; if a test case is hard to write, the requirement may be unclear.

Types of checks: Validity, Consistency, Completeness, Realism, Verifiability

Requirements Change — Why Requirements Evolve

  • Reason 1: Business/technical environment changes after installation (new hardware, changing priorities). E.g., installing RFID attendance and integrating with employee portal.
  • Reason 2: People who pay ≠ people who use. Customer imposes constraints from budget; after delivery, users need more features. E.g., EMS adding APR forms and PF statements.
  • Reason 3: Large systems have diverse user communities with conflicting priorities. Final requirements are always a compromise.

Requirements Management

Managing changing requirements throughout development. New requirements always emerge during development and after deployment.

Requirements Management Planning
  • Requirements Identification — Each requirement gets a unique ID for cross-referencing
  • Change Management Process — Activities to assess impact and cost of changes
  • Traceability Policies — Define relationships between requirements and design decisions
  • Tool Support — Spreadsheets, databases, dedicated requirements management tools
Requirements Change Management Process
  1. Identify the proposed change
  2. Assess the impact of the change (what does it affect?)
  3. Estimate cost of the change
  4. Decide whether to accept or reject the change
  5. If accepted: update requirements and notify stakeholders

Agile's Take on Requirements Engineering

In agile, detailed system requirements are considered waste of time.

  • Concurrent requirement elicitation
  • No formal requirement document
  • Requirements expressed as user stories

🗂️ Work Breakdown Structure (WBS)

WBS.pptx

What is WBS?

A hierarchical decomposition of the total scope of work to be carried out by the project team to accomplish project objectives and create required deliverables.

Scope and Planning = defining a project's deliverables, goals, tasks, deadlines, costs, and how to develop the software.

Why is WBS Needed?

  • Activity identification
  • Correct estimation of effort and cost
  • Identify related activities using PERT
  • Creating schedules
  • Minimizes risk of project failure

When to Develop WBS?

⏰ Timing
After finalizing project scope and before project estimation. (Scope → WBS → Estimation)

Levels in WBS

0
Level 0 — Project TitleThe root node; the entire project.
1
Level 1 — Control AccountsHighest-level breakdown into 4–7 nodes (major phases/deliverables).
2
Level 2 — Work PackagesSub-divisions of control accounts; manageable units of work.
3
Level 3 — Tasks/ActivitiesSpecific activities to be performed; small enough for accurate estimation.

Follow a standard numbering scheme (e.g., 1.0, 1.1, 1.1.1).

WBS Guidelines

  • Must be easy to understand
  • Covers all objectives
  • Each element corresponds to a deliverable
  • Break down until you get accurate time and cost estimates
  • Should NOT be as detailed as a project plan
  • Can evolve over time
  • Each level should have no more than 7 items
  • Tools: GanttPro, Word, Excel, MS Project, Visio, Jira, ProjectManager.com

Example WBS — Redecorate Room

🏠 Sample WBS
0. Redecorate Room
  • 1. Prepare materials
    • 1.1 Buy paint
    • 1.2 Buy a ladder
    • 1.3 Buy brushes/rollers
    • 1.4 Buy wallpaper remover
  • 2. Prepare room
    • 2.1 Remove old wallpaper
    • 2.2 Remove detachable decorations
    • 2.3 Cover floor with old newspapers
    • 2.4 Cover electrical outlets with tape
    • 2.5 Cover furniture with sheets
  • 3. Paint the room
  • 4. Clean up the room
    • 4.1 Dispose/store leftover paint
    • 4.2 Clean brushes/rollers
    • 4.3 Dispose of old newspapers
    • 4.4 Remove covers

WBS Formats

  • Outlined (Indented) WBS — Hierarchical bullet list format (as shown above)
  • Graphical Tree WBS — Tree diagram like an org chart, visually shows hierarchy

📊 Software Project Estimation (Wideband Delphi)

Estimation_.pptx

What is Project Estimation?

Determining how much a project will take in terms of time, cost, money, and manpower.

Example: LMS Estimation
Without estimation: unrealistic deadlines, budget overruns, insufficient team.
With estimation:
  • Effort = 1,200 person-hours
  • Duration = 4 months
  • Cost = PKR 1,200,000
  • Team = 4 developers + 1 tester
→ Planning is now structured and controlled!

Why Estimation is Needed

  • Budget planning
  • Time scheduling
  • Resource allocation
  • Risk management
  • Stakeholder communication
  • Project feasibility analysis

Estimation Techniques (Overview)

  • COCOMO — Algorithmic model using lines of code
  • Wideband Delphi — Expert consensus-based estimation
  • Function Points Estimation — Based on functional complexity
  • LOC (Lines of Code) Estimation — Count anticipated code lines

For good estimation, you need: WBS | Deliverables | Clear effort estimate per task | List of assumptions | Team consensus that estimate is accurate

Assumptions in Estimation

Team members make assumptions to deal with incomplete information. Assumptions must be written down. If an assumption proves incorrect, it must be tracked back and corrected.

Delphi Technique (Foundation)

Gather input from multiple stakeholders, then create a final estimate.

1
Have a facilitator/coordinator
2
Create list of experts
3
Create 3–5 questionsE.g., "What's better for this project — Azure or AWS?"
4
Reach out for answers
5
Analyze & iterateRepeat until consensus

Wideband Delphi Process (6 Steps)

A repeatable estimation process using a team to generate and reach consensus on estimates. Team size: 3–7 project members + 1 moderator.

1
Choose the Team Project manager selects 3–7 team members and a moderator. Moderator should know the Delphi process but have NO stake in the outcome. Ideally, PM is part of estimation team (not moderator).
2
Kickoff Meeting Ensure everyone understands the Delphi process and has read the vision/scope document. Team brainstorms and writes assumptions. Team reviews the WBS. Team agrees on a unit of estimation (hours, days, etc.).
3
Individual Preparation Each team member independently estimates effort for each WBS task and notes any assumptions they needed to make.
4
Estimation Session Team comes together to reach consensus. Each member fills an estimation form. Moderator plots sum of effort from each form on a line. Multiple rounds of discussion and revision — individual numbers NOT discussed directly. Disagreements resolved by adding assumptions. Goal: Maximum convergence. Delta sheet shows variation between rounds.
5
Assemble Tasks PM collects final estimates and compiles the final task list, estimates, and assumptions.
6
Review Results PM reviews final task list with the estimation team.
🎯 Key Rules of Wideband Delphi
  • Individual estimate numbers are NEVER discussed directly — only tasks/assumptions are discussed
  • Session continues until estimates converge or team is unwilling to revise further
  • Moderator plots estimates on a line to show convergence visually
  • A "Delta Sheet" shows how much estimates vary after each round
