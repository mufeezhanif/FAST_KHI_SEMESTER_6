Introduction to AI

Foundational concepts, definitions, and the scope of Artificial Intelligence

What is AI?

AI is the field concerned with building agents that perceive their environment and take actions that maximize their chance of achieving their goals. AIMA defines AI on two dimensions: thought vs. behavior and human-like vs. rational.

Thinking Humanly

Cognitive modelling approach — the system thinks like a human. Requires understanding of how humans actually think (cognitive science).

Thinking Rationally

Laws of thought approach — using logic. Syllogisms, formal reasoning. Problem: not all knowledge is easily formalizable.

Acting Humanly

Turing Test approach — if it's indistinguishable from a human in conversation, it's intelligent. NLP, knowledge, reasoning, learning needed.

Acting Rationally

Rational agent approach (AIMA's focus). Do the right thing given available information. Does not require mimicking humans.

Rationality

A rational agent selects the action that is expected to maximize its performance measure, given its percept sequence and built-in knowledge.

Key insight: Rationality ≠ omniscience. An agent with limited knowledge can still be rational by doing the best it can with what it knows. Rationality is about expected outcomes, not guaranteed ones.

Turing Test

Proposed by Alan Turing (1950). A human interrogator chats (via text) with a human and a machine. If the interrogator cannot reliably tell which is which, the machine is said to exhibit intelligent behavior. Capabilities required: NLP, knowledge representation, automated reasoning, machine learning.

AI History Highlights

  • 1943 — McCulloch & Pitts: first neural network model
  • 1950 — Turing's "Computing Machinery and Intelligence"
  • 1956 — Dartmouth Conference; AI named as a field
  • 1960s–70s — Expert systems, LISP, search algorithms
  • 1980s — AI Winter, then resurgence with expert systems
  • 1997 — Deep Blue defeats Kasparov in chess
  • 2012+ — Deep learning revolution

Agents & Environments

PEAS framework, types of agents, and environment classifications

PEAS Framework

To design an agent, first specify the task environment using PEAS: Performance measure, Environment, Actuators, Sensors.

P
Performance Measure

The criteria used to evaluate the agent's success. What does "doing well" mean? e.g., for a taxi: safe, fast, legal, comfortable, maximizes profit.

E
Environment

The external world the agent interacts with. e.g., for a taxi: roads, traffic, pedestrians, customers, weather.

A
Actuators

The mechanisms by which the agent acts on the environment. e.g., for a taxi: steering wheel, accelerator, brakes, horn, indicator.

S
Sensors

The means by which the agent perceives its environment. e.g., for a taxi: cameras, sonar, speedometer, GPS, odometer.

PEAS Example: Medical Diagnosis System

Component Description
Performance Healthy patient, minimized costs, successful diagnosis, legal compliance
Environment Patient, hospital, lab results, EHR database
Actuators Screen output (diagnosis, test orders, prescriptions)
Sensors Keyboard input (symptoms), lab results input
Types of Agents
1. Simple Reflex Agent Condition-Action Rules

Acts based only on the current percept, ignoring history. Uses a set of condition–action rules: IF condition THEN action.

Limitation: Fails in partially observable environments. No memory = no ability to deal with sequences. Works only if the correct action can be decided from current percept alone.

Example: Thermostat — if temperature < setpoint, turn on heater.

2. Model-Based Reflex Agent Internal State

Maintains an internal state (model of the world) to handle partial observability. The model describes: (a) how the world evolves independently; (b) how the agent's actions affect the world.

Rule selection: condition-action rules using BOTH current percept AND internal state.

Example: Self-driving car that remembers what was behind it when checking mirrors.

3. Goal-Based Agent Goal + Search/Planning

Knows what goal it is trying to achieve. Uses search and planning to find action sequences that lead to goal states.

More flexible than reflex agents — if environment changes, the agent can replan. Requires considering the future: "what will happen if I do X?"

Example: Route-planning GPS — knows destination (goal), plans the best route.

4. Utility-Based Agent Utility Function

Uses a utility function that maps states to real numbers, expressing how "happy" the agent is in that state. Maximizes expected utility.

Goals are binary (reached/not reached). Utility allows trade-offs between competing goals and handling uncertainty gracefully.

Example: Taxi choosing between a fast but risky route vs. a slow but safe route — weighs speed, safety, comfort.

5. Learning Agent 4 Components

Can improve its performance over time. Has four components:

  • Learning element — makes improvements based on feedback
  • Performance element — selects actions (the agent itself)
  • Critic — evaluates how well the agent is doing against a performance standard
  • Problem generator — suggests exploratory actions to gain new experience

Search Algorithms

Uninformed (blind) and Informed (heuristic-guided) search strategies

Problem Formulation

A search problem is defined by:

  • Initial state — where the agent starts
  • Actions — what the agent can do
  • Transition model — result of each action
  • Goal test — how to check if goal is reached
  • Path cost — numeric cost of each path

A solution is a sequence of actions from initial state to a goal state. An optimal solution has the lowest path cost among all solutions.

Key Performance Metrics

  • Completeness — Is it guaranteed to find a solution if one exists?
  • Optimality — Does it find the optimal (least cost) solution?
  • Time complexity — How many nodes are generated?
  • Space complexity — How many nodes are stored in memory?

Notation: b = branching factor, d = depth of shallowest goal, m = maximum depth, l = depth limit.

The Graph Used for Dry Runs

Uninformed (Blind) Search

These strategies have no additional information about states beyond the problem definition. They explore blindly using only the structure of the search tree.

BFS Breadth-First Search Uninformed

Idea: Expand the shallowest unexpanded node. Uses a FIFO queue (frontier). Explores level by level.

How it Works

1
Add the initial state to the queue.
2
Dequeue the front node. If it's the goal, return the solution.
3
Expand it — add all successors to the back of the queue (if not already visited).
4
Repeat from step 2 until queue is empty (failure) or goal found.

DFS Depth-First Search Uninformed

Idea: Always expand the deepest node first. Uses a LIFO stack (or recursion). Goes as deep as possible before backtracking.

How it Works

1
Push initial state onto the stack.
2
Pop the top node. If goal, return solution.
3
Push all successors onto the stack (right to left so left is explored first).
4
Repeat until stack empty or goal found.

DLS Depth-Limited Search Uninformed

Idea: DFS with a pre-set depth limit l. Nodes at depth l are treated as if they have no successors. Fixes DFS's infinite-loop problem in infinite graphs.

How it Works

Same as DFS but add a check: if current depth ≥ limit l, don't expand further. Returns failure if no goal found within limit, or cutoff if the limit was the reason for failure (goal may exist deeper).

IDS Iterative Deepening Search Uninformed

Idea: Run DLS with increasing depth limits l = 0, 1, 2, 3, ... until goal is found. Best of BFS + DFS: completeness & optimality of BFS, linear space of DFS.

How it Works

Each iteration is a fresh DLS. Nodes are re-generated rather than stored. The "wasted" work of repeated generations is small because most nodes are in the deepest layer: with b=10, 90% of nodes are in the last level, so total overhead is ~b/(b-1) = only ~11% extra.

UCS Uniform Cost Search Uninformed

Idea: Expand the node with the lowest cumulative path cost g(n). Uses a priority queue ordered by g(n). Unlike BFS which finds shallowest goal, UCS finds cheapest-cost goal. BFS is a special case of UCS when all costs are equal.

How it Works

1
Add initial state to priority queue with cost 0.
2
Dequeue node with lowest g(n). If goal, return path.
3
Expand it — add successors with updated costs g(child) = g(parent) + edge_cost.
4
If a node is already in queue with higher cost, update it (decrease-key).
Important: UCS applies the goal test when a node is dequeued (expanded), NOT when it is generated. This ensures optimality.

BiDs Bidirectional Search Uninformed

Idea: Run two simultaneous BFS searches — one forward from the start, one backward from the goal. Stop when the two frontiers meet. The key insight: bd/2 + bd/2 ≪ bd.

How it Works

1
Forward frontier starts from Start. Backward frontier starts from Goal.
2
Alternate expanding both frontiers (expand the smaller one).
3
After each expansion, check if the new nodes are in the OTHER frontier.
4
When a node appears in both frontiers — the two searches have met. Reconstruct path.
Challenge: The backward search requires computing predecessors — which states lead TO a given state. Also, finding a path when frontiers meet doesn't guarantee it's optimal; additional checking may be needed.

Intersection found at C. Reconstruct: Forward path: A → C Backward path: C → F → G Full path: A → C → F → G ✓ Savings: Explored d/2 = 1.5 levels each side instead of d=3 levels total.

Informed (Heuristic) Search

These strategies use problem-specific knowledge — a heuristic function h(n) — to guide search toward the goal more efficiently.

Heuristic h(n) = estimated cost from node n to goal.
Admissibility: h(n) ≤ h*(n) for all n — heuristic never overestimates.
Consistency (monotonicity): h(n) ≤ cost(n,n') + h(n') — satisfies triangle inequality. Consistency ⟹ admissibility.

Heuristic Example for Reference Graph

Example Heuristic Values h(n) = estimated dist to G
h(A)=4, h(B)=5, h(C)=2, h(D)=6, h(E)=7, h(F)=1, h(G)=0
GBFS Greedy Best-First Search Informed

Idea: Always expand the node that appears closest to the goal according to the heuristic. Priority queue ordered by h(n) alone. "Greedy" — myopically tries to get to the goal fast, ignoring actual path cost so far.

f(n) = h(n) (heuristic estimate only)

How it Works

Same as UCS but the priority queue is sorted by h(n) instead of g(n). Expands whatever node looks closest to goal, not necessarily the one with the cheapest actual cost so far.

A* Search Informed · Gold Standard

Idea: Combines actual cost g(n) AND heuristic h(n). Best of UCS and GBFS. Expands node with lowest f(n) = g(n) + h(n) — total estimated cost of cheapest solution through n.

f(n) = g(n) + h(n)

g(n) = actual cost so far | h(n) = estimated cost to goal

Why A* is Optimal

If h(n) is admissible (tree search) or consistent (graph search), A* is both complete and optimal. The proof: if a suboptimal goal G₂ is in the queue, there must be a frontier node n on the optimal path with f(n) ≤ C*. Since G₂ is suboptimal f(G₂) > C*, so n will always be expanded before G₂.

IDA* Iterative Deepening A* Informed · Memory Efficient

Idea: Apply iterative deepening strategy to A*. Instead of limiting by depth, limit by f-cost threshold. Each iteration is a depth-first search that prunes nodes with f(n) > current threshold. After each iteration, raise threshold to the minimum f-cost that was pruned.

How it Works

1
Set threshold = h(start).
2
Run DFS, pruning any node where f(n) = g(n) + h(n) > threshold.
3
If goal found, return solution. Otherwise, new threshold = min f-value that was cut off in this iteration.
4
Repeat from step 2 with new threshold.

SMA* Simplified Memory-Bounded A* Informed · Memory Bounded

Idea: A* but limited to a fixed memory of N nodes. When memory is full, drop the worst node (highest f-value) and back up that f-value to its parent (so we can regenerate it later if needed).

How it Works

1
Run A* normally until memory (N nodes) is full.
2
When full and a new node needs to be added, drop the node with the highest f-cost from memory.
3
Store the dropped node's f-cost in its parent — the parent now knows "via this child, the best reachable cost is X."
4
Continue. If parent is re-expanded in the future, it regenerates the dropped child.
Key concept — Backed-up f values: When a node is forgotten, its best f-cost is stored at the parent. The parent's f-value becomes max(f(parent), min f of forgotten child). This ensures we don't permanently lose track of good paths.

Local Search & Optimization

Hill Climbing, Gradient Descent, Beam Search, Simulated Annealing, Genetic Algorithms

What is Local Search?

Local search algorithms operate on a single current state and move to neighboring states. They do NOT maintain a path from the start — only the current state matters. They are used when the goal is to find a good state (optimization), not a path to it.

Key properties: Use very little memory (constant space). Can find solutions in infinite/continuous state spaces. Complete search algorithms aren't suitable when state space is huge. The downside: they can get stuck in local optima.

State Space Landscape

Imagine a landscape where the x-axis is the state, y-axis is objective (or value). We want to find the global maximum (highest hill). Problems:

  • Local maximum — peak that is not globally best. A greedy climber gets stuck here.
  • Plateau — flat region where all neighbors have the same value. Hard to know which way to go.
  • Ridge — slope in diagonal direction; going left/right or up/down doesn't help.
Hill Climbing (Steepest Ascent) Local Search

Idea: Start at a random state. Always move to the highest-value neighbor. Stop when no neighbor is better (local maximum). Like climbing a hill in the dark — always go uphill.

Algorithm (Pseudocode)

Hill Climbing Pseudocode
function HILL-CLIMBING(problem): current ← problem.initial_state loop: neighbor ← highest-valued successor of current if VALUE(neighbor) ≤ VALUE(current): return current ← local maximum found current ← neighbor

Variants of Hill Climbing

  • Steepest-Ascent — examine all neighbors, pick the best one
  • Stochastic — choose randomly among uphill moves (weighted by steepness)
  • First-choice — pick the first random neighbor that is better (useful when many neighbors)
  • Random-restart — restart from random state when stuck. With k restarts, probability of finding global max increases. Each restart: O(steps). Total: O(k × steps)

Gradient Descent Local Search · Continuous

Idea: The continuous-space analog of hill climbing. Instead of comparing discrete neighbors, compute the gradient (derivative) of the objective function and take a step in the direction of steepest descent (to minimize) or ascent (to maximize).

θ_new = θ_old − α · ∇J(θ)

α = learning rate | ∇J(θ) = gradient of loss function

How it Works

1
Start at an initial point θ (randomly or by convention).
2
Compute gradient ∇J(θ) — the direction of steepest ascent of the cost.
3
Update θ by stepping in the opposite direction (downhill for minimization).
4
Repeat until convergence (gradient ≈ 0 or change is very small).

Learning Rate α — Critical Parameter

  • Too small α: Very slow convergence, many iterations needed.
  • Too large α: Overshoots the minimum, may diverge (oscillate or explode).
  • Just right: Converges smoothly. Often found by learning rate scheduling or adaptive methods (Adam, RMSProp).
📋 Dry Run — Gradient Descent on J(θ) = (θ−3)², α=0.5

Beam Search Local k-Best Local Search

Idea: Keep track of k states (beam width) at a time instead of 1. At each step, generate all successors of all k states, then keep only the k best. A middle ground between BFS (all states) and Hill Climbing (1 state).

How it Works

1
Start with k randomly generated states.
2
Generate all successors of all k states (k × b successors total).
3
Select the k best successors from this pool.
4
Repeat until goal is found or no improvement.

Stochastic Beam Search: Instead of keeping the k best deterministically, choose k successors probabilistically — states with better heuristic values have higher chance of selection. Introduces diversity, helps escape local optima. Similar to natural selection.
Beam Search weakness: All k states can quickly concentrate in the same region (lack of diversity). One branch dominates and others die out — effectively becomes hill climbing. Stochastic variant helps.
Simulated Annealing Local Search · Probabilistic

Idea: Inspired by the metallurgical process of annealing (slow cooling of metal to reduce defects). Occasionally accept worse moves with a probability that decreases over time (as "temperature" T drops). This allows escape from local optima early on, then converges as T→0.

The Acceptance Probability

P(accept worse move) = eΔE / T

ΔE = VALUE(new) - VALUE(current) < 0 for a worse move. As T decreases, probability of accepting worse moves decreases. At T=0, only improvements are accepted (becomes hill climbing).

Algorithm

Simulated Annealing Pseudocode
function SIMULATED-ANNEALING(problem, schedule): current ← problem.initial_state for t = 1 to ∞: T ← schedule(t) // temperature decreases over time if T = 0: return current next ← random successor of current ΔE ← VALUE(next) - VALUE(current) if ΔE > 0: current ← next // always accept improvements else: with probability e^(ΔE/T): current ← next // maybe accept worse
📋 Dry Run — Simulated Annealing Example
Problem: f(x) to maximize. Current x=2, f(2)=10, T=100 Schedule: T decreases by factor 0.9 each step. Step 1 (T=100): Pick random neighbor x=3, f(3)=8. ΔE=8-10=-2 (worse) P = e^(-2/100) = e^(-0.02) ≈ 0.98 Random draw: 0.45 < 0.98 → ACCEPT (move to worse state!) Current: x=3, f=8 Step 2 (T=90): Pick random neighbor x=4, f(4)=15. ΔE=15-8=7 (better!) → ALWAYS ACCEPT. Current: x=4, f=15 Step 3 (T=81): Pick random neighbor x=3, f(3)=8. ΔE=8-15=-7 (worse) P = e^(-7/81) = e^(-0.086) ≈ 0.917 Random draw: 0.95 > 0.917 → REJECT. Stay at x=4. ... (as T→0, only improvements accepted) ... Late stage (T=0.01): ΔE=-2: P = e^(-200) ≈ 0. REJECT. Algorithm essentially pure hill climbing. Eventually converges to (near) global maximum. ✓

Applications: VLSI design, travelling salesman problem, scheduling, protein structure prediction — anywhere the search space is huge and rugged.
Genetic Algorithm Local Search · Evolutionary

Idea: Inspired by natural evolution. Maintain a population of states (chromosomes). Combine good states using crossover and introduce variation via mutation. Over generations, the population evolves toward better solutions.

Key Components

  • Chromosome (Individual) — A state encoded as a string (usually binary). Each character is a gene.
  • Population — A set of k individuals (chromosomes).
  • Fitness function — Evaluates how good each individual is. Higher fitness = more likely to reproduce.
  • Selection — Choose parents based on fitness (e.g., roulette wheel, tournament selection).
  • Crossover (Recombination) — Combine two parents at a random cut-point to create offspring.
  • Mutation — Randomly flip a gene with small probability. Prevents premature convergence.

Algorithm Steps

1
Initialize a random population of k chromosomes.
2
Evaluate each chromosome using the fitness function.
3
Select pairs of parents — fitter individuals selected more often.
4
Crossover — for each pair, pick a random point and swap tails to create two children.
5
Mutate — randomly alter a gene in each child with low probability (e.g., 1%).
6
Replace old population with new generation of children.
7
Repeat from step 2 until termination (goal fitness, max generations, convergence).

Why Genetic Algorithms work: The Schema theorem — short, low-order building blocks (schemas) with above-average fitness propagate exponentially across generations. Good partial solutions combine to form even better solutions (building block hypothesis).
GA vs Simulated Annealing: Both are metaheuristics that escape local optima. SA uses a single solution with random restarts. GA uses a population — maintains diversity naturally. GA can implicitly parallelize (multiple good regions explored). SA has simpler implementation.
Parameter tuning matters: Mutation rate too high → random search. Too low → premature convergence. Population too small → lack of diversity. Too large → slow. Crossover probability: typically 60–90%. Mutation probability: typically 1–5%.

BFS: FIFO queue, finds shallowest, complete + optimal (unit cost), memory hog O(b^d).
DFS: LIFO stack, may not terminate on infinite graphs, O(b·m) space (great!).
IDS: Best uninformed in practice. BFS properties + DFS space. Re-generates nodes.
UCS: BFS with variable costs. Orders by g(n). Apply goal test at expansion.
A*: f=g+h. Optimal if h is admissible. Dominates all other optimal search with same h.
IDA*: A* with DFS + f-cost cutoff threshold. Linear space. Repeats nodes.
GBFS: Only h(n). Fast but not optimal, not complete.
Hill Climbing: No memory, always uphill, gets stuck at local maxima.
Simulated Annealing: Probabilistically accepts worse states. Temperature decreases over time.
Genetic Algorithm: Population, selection, crossover, mutation. Inspired by evolution.
